本项目遵循 [Google 开发者文档风格指南](https://developers.google.com/style) 并使用 [Diataxis](https://diataxis.fr/) 来组织内容。仅列出项目特定的约定。

## 项目特定规则

| 规则 | 规范 |
| -------------------------------- | ---------------------------------------- |
| 无水平分割线 (`---`) | 会打断阅读流 |
| 无 `####` 标题 | 使用粗体文本或 Admonitions 代替 |
| 无 "Related" 或 "Next:" 章节 | 侧边栏处理导航 |
| 无深层嵌套列表 | 分成章节代替 |
| 非代码内容不使用代码块 | 对话示例使用 Admonitions |
| 标注不使用粗体段落 | 使用 Admonitions 代替 |
| 每节最多 1-2 个 Admonitions | 教程的主要章节允许 3-4 个 |
| 表格单元格 / 列表项 | 最多 1-2 句话 |
| 标题预算 | 每个文档 8-12 个 `##`；每节 2-3 个 `###` |

## Admonitions (Starlight 语法)

```md
:::tip[Title]
快捷方式，最佳实践
:::

:::note[Title]
上下文，定义，示例，先决条件
:::

:::caution[Title]
注意事项，潜在问题
:::

:::danger[Title]
仅限严重警告 — 数据丢失，安全问题
:::
```

### 标准用途

| Admonition | 用途 |
| ------------------------ | ----------------------------- |
| `:::note[Prerequisites]` | 开始前的依赖项 |
| `:::tip[Quick Path]`     | 文档顶部的太长不看 (TL;DR) 摘要 |
| `:::caution[Important]`  | 关键注意事项 |
| `:::note[Example]`       | 命令/响应示例 |

## 标准表格格式

**阶段 (Phases):**

```md
| Phase | Name     | What Happens                                 |
| ----- | -------- | -------------------------------------------- |
| 1     | Analysis | 头脑风暴，研究 *(可选)*            |
| 2     | Planning | 需求 — PRD 或技术规范 *(必需)* |
```

**命令 (Commands):**

```md
| Command      | Agent   | Purpose                              |
| ------------ | ------- | ------------------------------------ |
| `brainstorm` | Analyst | 头脑风暴一个新项目             |
| `prd`        | PM      | 创建产品需求文档 (PRD) |
```

## 文件夹结构块

在 "你完成了什么 (What You've Accomplished)" 章节中展示：

````md

your-project/

│   ├── PRD.md                     # 你的需求文档

```
````

## 教程结构 (Tutorial Structure)

```text
1. 标题 + 钩子 (1-2 句话描述结果)
2. 版本/模块通知 (信息或警告 admonition) (可选)
3. 你将学到什么 (结果的项目符号列表)
4. 先决条件 (info admonition)
5. 快速路径 (tip admonition - TL;DR 摘要)
6. 理解 [主题] (步骤前的上下文 - 阶段/agents 表格)
7. 安装 (可选)
8. 步骤 1: [第一个主要任务]
9. 步骤 2: [第二个主要任务]
10. 步骤 3: [第三个主要任务]
11. 你完成了什么 (摘要 + 文件夹结构)
12. 快速参考 (命令表格)
13. 常见问题 (FAQ 格式)
14. 获取帮助 (社区链接)
15. 关键要点 (结尾的 tip admonition)
```

### 教程检查清单

- [ ] 钩子用 1-2 句话描述结果
- [ ] 存在 "你将学到什么" 章节
- [ ] 先决条件在 admonition 中
- [ ] 顶部有快速路径 TL;DR admonition
- [ ] 阶段、命令、agents 的表格
- [ ] 存在 "你完成了什么" 章节
- [ ] 存在快速参考表格
- [ ] 存在常见问题章节
- [ ] 存在获取帮助章节
- [ ] 结尾有关键要点 admonition

## How-To 结构

```text
1. 标题 + 钩子 (一句话: "使用 `X` workflow 来...")
2. 何时使用 (场景的项目符号列表)
3. 何时跳过 (可选)
4. 先决条件 (note admonition)
5. 步骤 (编号的 ### 子章节)
6. 你会得到什么 (产生的输出/artifacts)
7. 示例 (可选)
8. 提示 (可选)
9. 下一步 (可选)
```

### How-To 检查清单

- [ ] 钩子以 "使用 `X` workflow 来..." 开头
- [ ] "何时使用" 有 3-5 个要点
- [ ] 列出先决条件
- [ ] 步骤是编号的 `###` 子章节，使用动作动词
- [ ] "你会得到什么" 描述输出 artifacts

## 解释与说明结构 (Explanation Structure)

### 类型

| 类型 | 示例 |
| ----------------- | ---------------------------- |
| **索引/落地页** | `core-concepts/index.md`     |
| **概念**       | `what-are-agents.md`         |
| **特性**       | `quick-flow.md`              |
| **哲学**    | `why-solutioning-matters.md` |
| **FAQ**           | `brownfield-faq.md`          |

### 通用模板

```text
1. 标题 + 钩子 (1-2 句话)
2. 概览/定义 (它是什么，为什么重要)
3. 关键概念 (### 子章节)
4. 对比表格 (可选)
5. 何时使用 / 何时不使用 (可选)
6. 图表 (可选 - mermaid, 每个文档最多 1 个)
7. 下一步 (可选)
```

### 索引/落地页

```text
1. 标题 + 钩子 (一句话)
2. 内容表格 (带描述的链接)
3. 开始使用 (编号列表)
4. 选择你的路径 (可选 - 决策树)
```

### 概念解释

```text
1. 标题 + 钩子 (它是什么)
2. 类型/分类 (### 子章节) (可选)
3. 关键差异表格
4. 组件/部分
5. 你应该使用哪个？
6. 创建/自定义 (指向 how-to 指南)
```

### 特性解释

```text
1. 标题 + 钩子 (它做什么)
2. 快速概览 (可选 - "适合:", "耗时:")
3. 何时使用 / 何时不使用
4. 它是如何工作的 (可选 mermaid 图表)
5. 关键优势
6. 对比表格 (可选)
7. 何时进阶/升级 (可选)
```

### 哲学/原理文档

```text
1. 标题 + 钩子 (原则)
2. 问题
3. 解决方案
4. 关键原则 (### 子章节)
5. 优势
6. 何时适用
```

### 解释检查清单

- [ ] 钩子说明文档解释了什么
- [ ] 内容在可扫描的 `##` 章节中
- [ ] 3+ 选项时使用对比表格
- [ ] 图表有清晰的标签
- [ ] 包含指向 how-to 指南的链接以解决程序性问题
- [ ] 每个文档最多 2-3 个 admonitions

## 参考结构 (Reference Structure)

### 类型

| 类型 | 示例 |
| ----------------- | --------------------- |
| **索引/落地页** | `workflows/index.md`  |
| **目录**       | `agents/index.md`     |
| **深度剖析**     | `document-project.md` |
| **配置** | `core-tasks.md`       |
| **术语表**      | `glossary/index.md`   |
| **综合指南** | `bmgd-workflows.md`   |

### 参考索引页

```text
1. 标题 + 钩子 (一句话)
2. 内容章节 (每个分类 ##)
   - 带链接和描述的项目符号列表
```

### 目录参考

```text
1. 标题 + 钩子
2. 项目 (每个项目 ##)
   - 简短描述 (一句话)
   - **命令:** 或 **关键信息:** 平铺列表
3. 通用/共享 (## 章节) (可选)
```

### 项目深度剖析参考

```text
1. 标题 + 钩子 (一句话目的)
2. 快速概览 (可选 note admonition)
   - 模块, 命令, 输入, 输出作为列表
3. 目的/概览 (## 章节)
4. 如何调用 (代码块)
5. 关键章节 (每个方面 ##)
   - 子选项使用 ###
6. 注意/警告 (tip 或 caution admonition)
```

### 配置参考

```text
1. 标题 + 钩子
2. 目录 (如果 4+ 项目则跳转链接)
3. 项目 (每个配置/任务 ##)
   - **粗体摘要** — 一句话
   - **何时使用:** 项目符号列表
   - **它是如何工作的:** 编号步骤 (最多 3-5 步)
   - **输出:** 预期结果 (可选)
```

### 综合参考指南

```text
1. 标题 + 钩子
2. 概览 (## 章节)
   - 展示组织的图表或表格
3. 主要章节 (每个阶段/分类 ##)
   - 项目 (每个项目 ###)
   - 标准化字段: 命令, Agent, 输入, 输出, 描述
4. 下一步 (可选)
```

### 参考检查清单

- [ ] 钩子说明文档参考了什么
- [ ] 结构匹配参考类型
- [ ] 项目贯穿始终使用一致的结构
- [ ] 结构化/对比数据使用表格
- [ ] 包含指向解释文档的链接以获得概念深度
- [ ] 最多 1-2 个 admonitions

## 术语表结构 (Glossary Structure)

Starlight 从标题生成右侧 "本页内容" 导航：

- 分类作为 `##` 标题 — 出现在右侧导航中
- 术语在表格中 — 紧凑的行，不是单独的标题
- 无内联目录 (TOC) — 右侧边栏处理导航

### 表格格式

```md
## 分类名称

| 术语 | 定义 |
| ------------ | ---------------------------------------------------------------------------------------- |
| **Agent**    | 具有特定专业知识的专用 AI 角色，指导用户完成工作流。 |
| **Workflow** | 多步骤引导过程，编排 AI Agent 活动以产生交付物。 |
```

### 定义规则

| 做 (Do) | 不做 (Don't) |
| ----------------------------- | ------------------------------------------- |
| 以它是什么或做什么开头 | 以 "This is..." 或 "A [term] is..." 开头 |
| 保持在 1-2 句话 | 写多段解释 |
| 在单元格中加粗术语名称 | 对术语使用纯文本 |

### 上下文标记

在定义开始处添加斜体上下文，用于有限范围的术语：

- `*仅限 Quick Flow。*`
- `*BMad Method/企业级。*`
- `*阶段 N。*`
- `*BMGD。*`
- `*Brownfield。*`

### 术语表检查清单

- [ ] 术语在表格中，不是单独的标题
- [ ] 术语在分类内按字母顺序排列
- [ ] 定义 1-2 句话
- [ ] 上下文标记使用斜体
- [ ] 单元格中术语名称加粗
- [ ] 无 "A [term] is..." 定义

## FAQ 章节

```md
## 常见问题 (Questions)

- [我是否总是需要架构？](#do-i-always-need-architecture)
- [以后可以更改计划吗？](#can-i-change-my-plan-later)

### 我是否总是需要架构？

仅限 BMad Method 和企业级轨道。Quick Flow 跳至实现。

### 以后可以更改计划吗？

可以。SM Agent 有一个 `correct-course` workflow 用于处理范围变更。

**有问题在这里没找到答案？** [提交 Issue](/docs/zh-cn/...) 或在 [Discord](/docs/zh-cn/...) 中提问。
```

## 验证命令

在提交文档更改之前：

```bash
npm run docs:fix-links            # 预览链接格式修复
npm run docs:fix-links -- --write # 应用修复
npm run docs:validate-links       # 检查链接存在
npm run docs:build                # 验证无构建错误
```
